应用层协议原理

网络应用程序体系结构（application architecture）

定义：由应用程序研发者设计，规定如何在各种end system中组织该应用程序

主要有两种体系结构：客户——服务器体系结构（client-server architecture）， 对等体系结构（P2P）

client-server architecture。client之间不能通信，可能间歇性的与server连接，可能有着动态IP地址。Server，一直在线，一般静态IP地址，扩展的数据中心。

一般来说server会形成data center以保障承受大量的数据。

P2P体系结构（P2P architecture），有一个特点，对于data cneter的专用服务器有最小的依赖。

与client-server architecture不同，每个end system彼此既是server又是client，这也就说它们彼此之间可以随意连接。同时，它们并不是实时在线。它们有一个很特殊的特征 自拓展性(self-scalability) 。self-scalibilty带来了新的服务能力和新的需求。

进程通信

我们知道一个end system一般会有多个程序，那么它们之间是如何通信的。严谨点说程序之间的通信实际上是进程（process）之间的通信。不同之间的process通过message进行通信。

在一对process通信的过程中，发起通信的process被标记为client，在会话开始时等待连接的进程是server。比如，web浏览器和web服务器

我们知道process存在在application layer，所以我们进行不同end system之间的通信时，必须要经过下层的layer。process通过一个称为 套接字（socket） 的软件接口向网络发送报文和从网络接收报文，简单地说与transmission layer进行通信。

由于socket建立在网络应用程序的可编程接口，因而又被称为应用程序与网络之间的 应用程序编程接口（Application Programming Interface，API）

当我们需要进行不同end system之间的process通信时，我们需要两个信息。
（1）end system的地址，IP地址
（2）在目的end system中指定接收Process的标识符。端口号（port number） 。

可供应用程序使用的运输服务

应用程序服务的需求可以分为：可靠数据传输、吞吐量、定时、安全性

可靠数据传输（reliable data transfer） 确保应用程序的一端发送的数据正确、完全地交付给该应用程序的另一端。

当transmission layer无法提供reliable data transfer时，有可能被 容忍丢失的应用（loss-tolerant application）所接受。

吞吐量，就是发送方process能够向接收方Process交付比特的速率。

这就产生一个需求，transmission layer可以以每种特定的速率提供确保的吞吐量。对吞吐量有需求的应用称为 带宽敏感的应用（bandwidth-senstive application） 。

定时，比如发送方注入进socket的每一个比特到达接收方的socket不迟于100ms。

安全性，transmission layer为应用程序提供一种或多种安全性服务。

因特网提供的运输服务

TCP，UDP

TCP，包括面向连接服务和可靠数据传输服务。

面向连接服务：TCP让client和server相互交换运输层控制信息，这被称为握手过程。握手后TCP Connection就在两个process的socket之间建立了。这个连接是全双工的。

可靠的数据传送服务：process 通信可以依靠TCP，无差别、按适当顺序支付所有发送的数据。

此外TCP还有拥塞控制。

为保障通信的安全，我们使用 安全套接字层（secure sockets layer,SSL）

UDP是一种不提供不必要服务的轻量级运输协议。它是一种不可靠的传输协议，不提供拥塞控制。

从上面的描述中我们可以看出现在的设计可以保障reliable data transfer 和 安全性（SSL），同时现在可以满足时间敏感应用提供满意服务，但是不能提供任何定时和带宽保证。

应用层协议（application layer protocol）

application layer protocol定义了运行在不同end system上的应用程序process如何互相传递message。

定义了，交换的报文类型，比如请求报文和响应报文。
各种报文类型的语法。
字段的语义，
确定一个进程何时以及如何发送报文，对报文的响应规则

HTTP

概述

Web的应用层协议是 超文本传输协议（HyperText Transfer Protocol，HTTP）

Web页面（Web page）是由对象组成的。一个object可以是一个文件、图片等。

HTTP定义了Web客户向Web Server请求Web page的方式，以及Server向client发送Web page的方式。

HTTP使用TCP作为他的支撑传输协议。

HTTP协议不用担心数据丢失，也不关注TCP从网络数据丢失和乱序故障中恢复的细节。这也是protocol slack分层的好处。每一层只需要关注自己的事情，下层的事情无需关心，有点类似OOP

HTTP是一个 无状态协议（stateless protocol）

非持续连接和持续链接

非持续连接（non-persistent connection）：每个请求/响应对经一个单独的TCP连接发送。

持续连接连接（persistent connection）：所有请求以及响应经相同的TCP连接发送。

HTTP既能使用non-persistent connection和persistent connection

non-persistent connection HTTP的过程

（1）HTTP client在端口80向server发送一个TCP连接，client与server都有一个套接字与该连接相关联。

（2）HTTP client经它的套接字向server发送一个HTTP请求报文

（3）HTTP server从socket中接受这个请求，然后搜索请求的对象，并在HTTP响应报文中封装这个对象，从socket中发送出去。

（4）HTTP client process通知TCP断开此TCP连接。

（5)HTTP sever process关闭TCP连接。

往返时间（Round-Trip Time） ，该时间是指一个短分组从客户到服务器然后再返回client的时间。RTT包括processing delay、queuing delay、分组时延。

三次握手中前两个部分所消耗的时间占用一个RTT。完成三次握手的前两个部分后，客户结合三次握手的第三部分（确认）向该TCP连接发送一个HTTP请求报文。

简单地说总的响应时间是两个RTT加上服务器传输HTTP文件的时间

persistent connection HTTP，建立TCP连接后服务器做出响应后不会关闭TCP而是保持TCP连接此后请求与响应还能通过相同的连接进行传送。

一般来说，如果一条连接经过一定时间间隔仍未被使用，HTTP服务器就关闭该连接。

与HTTP的比较

HTTP从Web服务器向Web客户传送文件，SMTP从一个mail server向另一个mail server传送文件。

HTTP是一个 拉协议（pull protocol） ，简单地说从一个服务器中获取文件。

SMTP是一个 推协议（push protocol），见的说发送mail server把文件推向接收mail server。

第二个区别我们上面提及到了，SMTP需要每个报文采用7比特ascii码格式，HTTP没有这个限制。

第三个，HTTP将每个对象封装到自己的HTTP响应报文中，而SMTP则把所有的保温对象放在一个报文之中。

DNS：因特网的目录服务

因特网上的主机的标识方式为 主机名（hostname），比如www.facebook.com，但是我怎么看都像是域名，所以我查了一下，hostname就应该是我们传统意义上的主机名，有些疑惑。

主机还可以通过 IP地址（IP address）标识。

DNS提供的服务

域名系统（Domain Name System，DNS），可以将hostname与IP address进行转换。DNS是：（1）一个由分层的 DNS服务器（DNS server） 实现的分布式数据库；（2）一个使得主机可以查询分布式数据库的应用层协议。

DNS server一般是运行BIND（Berkeley Internet Name Domain）的软件的Unix机器。DNS运行在UDP上，端口为53。

DNS是一个应用层协议，主要原因在于（1）使用客户-服务器模式运行在通信的端系统之间（2）在通信的端系统之间通过传输层的端对端运输协议传送DNS报文。

但是，它还有很特殊一点，它不直接与用户打交道，而是为其他软件提供服务，即hostname转IP address。

当一个host上的browser访问一个violet.com网站，host主要经历了这些过程获得网站的Ip address。

（1）同一台用户Host上运行着DNS应用的客户端
（2）browser从上述的url中抽取hostname,然后将这个hostname转交给DNS客户端
（3）DNS client向DNS server发送一个包含hostname的请求。
（4）DNS client最终会收到一份回答报文，其中含有对应该hostname的IP address
（5）接收到IP address后，browser便可以直接访问ip address的80 port进行TCP 连接。

除了获得hostname的Ip address，DNS还提供了其他的service，

主机别名（host aliasing），当一个hostname非常难记时，比如1有rqwhrwqwq.violet.com，他可能有别名 EG.violet.com，这样方便记忆。这种情况下前者复杂的hostname又被称为 规范主机名（canonical hostname）。DNS可以获得host aliasing对应的canonical hostname以及Ip address。

邮件服务器别名（mail server aliasing），与host aliasing类似。

负载分配（load distribution），繁忙的站点可能有多个服务器，也就说一个站点有多个IP address，DNS将这些Ip address存储为一个集合，当host 向站点请求访问时，DNS server会返回这个站点的ip address集合，而host一般会向集合位置最前面的ip address进行http访问。而DNS利用这个特性，就可以将不繁忙的服务器的ip address放置在集合的前面，从而实现了load distribution。

DNS工作机理概述

DNS设计模式可以设计为整个Internet只使用一个DNS server，当然这肯定不可以，会出现 单点故障（a singal point of falilure） 、 通信容量（traffic volume）、 远距离的集中式数据库（distant centralized database）、 维护（maintenance）的问题。

为了解决这些问题，DNS使用了大量DNS server，他们以层次的分布在全世界范围内。

大致上有三种DNS server，根服务器，顶级域（Top-Level Domain，TLD）DNS服务器和权威服务器。它们的层次结构见下图。

root DNS server: 提供TLD的 ip address

TLD DNS server: 对于每个顶级域（如，com、org、net、edu）和所有国家的顶级域（cn、uk），都有TLD服务器（或服务器集群）。TLD服务器提供了权威DNS server 的ip address。

authoritative DNS server: 因特网上具有公共可访问主机（如Web服务器和邮件服务器）的每个组织机构必须提供公共可访问的DNS记录，这些记录将hostname映射为ip addres。

此外还有一个重要的DNS服务器 本地DNS服务器（local DNS Server） 。每一个ISP都有一台local DNS server。一般来说local dns server起着代理的作用，将client的DNS请求转发到DNS体系结构中。

DNS记录与报文

共同实现DNS分布式数据库的所有有DNS服务器存储了 资源记录（Resource Record，RR），RR提供了主机名向IP地址的映射。

RR是一个包含了下列字段的4元组，（Name，Value，Type，TTL）

TTL是该记录的生存时间。Name与Value取决于Type

Type=A，Name为主机名，Value为hostname对应的IP address。

Type=NS，则Name是个domain（如violetBK.github.io），而value是个知道如何获得该域中主机IP address的authoritative DNS server的hostname。比如（foo.com,dns.foo.com,NS）。

Type=CName，则value是个别名为Name对应的主机对应的canonical Name。

Type = MX，则value是个别名为Name的邮件服务器的canonical Name。为了获得邮件服务器的canonical Name，DNS应当请求一个MX记录，为了获得其他服务器的规范主机名，DNS客户应当请求CNAME记录。

DNS报文格式，如下图所示。

首先前12个字节为首部区域，identification，用于标识这个报文。flag中有若干标志。1bit的“查询/应答”标志位，指出该报文是查询（0），还是应答（1）。当DNS为authoritative DNS server时，1bit的“authoritative”标志被置1于应答报文。此外还可以设置“希望recursive/iteration”于请求，或者“recursive/iteration可用”于应答。此外还有4个表示数量的字段，来标志首部后4类数据区域出现的数量。

问题区域包含正在进行的查询信息。该区域包括，（1）名字字段，包含正在被查询的hostname。（2）类型字段，指出正在被查询的问题类型，比如A，MX。

来自DNS server的回答中，回答区域包含了对最初请求名字的RR。在应答报文的回答区域中可以有多个RR，也就说一个hostname可以有多个ip address。

权威区域包含了其他authoritative DNS server的record。

附加区域包含了其他有用信息，比如对于一个CName请求的回答报文的回答区域包含一条RR，该RR提供了host的canonical Name。该附加区域包含了一个A类型的RR，该RR提供了该host的canonical Name的IP address。

创建过网站的都会去尝试为自己的网站设置域名，注册登记机构（registrar）是一个商业实体，他延展该域名的唯一性，将该域名输入DNS数据库，并收取小部分费用。

当我们注册域名时，我们需要向registrar提供基本和辅助DNS server的名字和Ip address。该registrar会将一个NS类型和一个A类型的RR存储到TLD服务器中。NS记录，记录到你域名的DNS服务器的hostname，A记录记录authoritative DNS server的hostname和ip address。

P2P文件分发

client-server architecture有着服务器负担过大的问题。而在P2P文件分发中，每个对等体可以向任何对等体重新分发他已经收到的该文件的任何部分，从而在分发过程中协助该服务器。

在Clinet-Server architecture中我们下载完文件的时间为Dcs>=max{NF/us,F/dmin}

而在P2P中下载过程相对复杂。

首先下载速度还是会受到下载速率最慢的主机影响即F/dmin

其次，服务器的分发文件所需时间与client-server architectrue有所不同，我们只需要分发一份即可，即F/us

最后，便是P2P的独特之处，P2P的上载总能力为所有主机加起来在加上服务器，则所有文件的最小分发时间为NF/(us+u1+u2+…un)

最后Dp2p>= max{F/us,,F/dmin,,NF/(us + Sui)}

根据P2P文件分发有一个用于文件分发的协议BitTorrent（比特洪流）。我们这里简单说一下原理。

首先每一个torrent都一个基础设施节点，称之为 跟踪器（tracker）。当一个对等体加入某torrent时，他向tracker注册自己，并周期地通知该tracker它仍在该torrent中。通过这种方式，torrent跟踪参与在洪流中的对等体。

下面是一个Alice加入torrent的过程。

首先Alice加入torrent时，跟踪器随机从对等体集合中选择一个子集，并将其子集中的对等体的ip address分发给Alice。从而Alice与这些对等体建立TCP连接。我们将所有与Alice建立TCP连接的对等体称为“临近对等体”，随着时间的流逝临近对等体会随着波动。peers may come and go once peer has entire file, it may (selfishly) leave or (altruistically) remain in torrent

Alice会周期向临近对等体请求块列表（文件块），L个不同邻居就有了L个块列表，有了这个信息，ALice就可以向他还没有的快发出请求。

Alice需要做出以下的决定，首先她应当向她的邻居请求那些块，其次她应该向那些邻居请求。这里使用的时 最稀缺优先（rarest first）技术，在邻居中找她没有的块，并且寻找那些邻居中拥有量最少的块，由此保障每个块副本的数量均衡。

请求解决了，那么应答呢？Alice应当应答哪些请求呢？基本思想是，根据当前向她提供数据的速率的高低，给出优先权。书上的翻译有点辣鸡，我直接给出ppt的内容并自己概述一下。


Alice sends chunks to those four peers currently sending her chunks at highest rate 
other peers are choked by Alice (do not receive chunks from her)
re-evaluate top 4 every10 secs
every 30 secs: randomly select another peer, starts sending chunks
“optimistically unchoke” this peer
newly chosen peer may join top 4

Alice首先根据给她发送块的速率高低给她的邻居们排个序，这里她对4个最快的邻居 疏通（unchoke） 即发送数据。与此同时Alice每隔30s随机选择一个别的对等体并向他发送数据，由于Alice正在发送数据，所以Alice可能成为她选择对等体的4个unchoke之一，这样的话他就也会向Alice发送数据。如果这个块发送速率够快，那么他就有可能成为Alice中的上载速率前四，这样他们就持续交换数据，否则Alice过30s后就选择新的对等体。

在这个过程中Alice同时向5个对等体发送块。

这里我们再提一下另一个P2P应用–分布式散列表（DHT）,它是一种简单的数据库，其数据库记录分布在一个p2p系统的多个对等体上。

视频流和内容分发网

因特网视频

视频的一个重要特征是它可以被压缩，因而可用比特率来衡量视频质量。低质量视频100kps，流式高分辨率率电影超过3Mbps，4k流式展望超过10Mbps。

对于流式视频最重要的性能度量是平均端到端吞吐量。

HTTP流和DASH

当用户要看视频时，客户与服务器之间创建一个TCP连接并发送对该HTTP GET请求。服务器要以尽可能快的速率在HTTP响应报文中发送该视频文件。而客户，字节被手机在客户应用缓存中，当缓存中的字节数量超过预先设定的门限，客户应用程序就开始播放。而流式视频应用会用程序周期性的从客户应用程序缓存中抓取真，对这些真进行解压缩并在用户屏幕上播放。

下面我们介绍一个可以根据用户带宽大小自动选择比特率的 经HTTP的动态适应性流（Dynamic Adapative Streaming over HTTP，DASH）。

HTTP服务器还有一个 告示文件（mainifest file） 为每个视频版本提供一个URL和比特率。

内容分发网

为了应对向分布于全世界的用户分发巨量视频数据的挑战，几乎所有视频公司都使用了 内容分发网（Content Distribution Network,CDN）。

CDN有两个不同的服务器安置原则，

深入 ，深入到ISP的接入网（即靠近end system的ISP），目的时靠近端用户，通过减少端用户和CDN集群之间的链路和路由器数量，改善用户感到的延迟和吞吐量。

邀请做客 ，通过在少量的关键位置建造大集群邀请ISP做客，CDN通常将他们的集群防止在因特网交换点（IXP）。

一旦一个CDN集群准备完成，它可以跨进群复制内容，一般根据CDN集群的服务类群确认。

当一个主机请求一个特定的视频（由URL标识），CDN必须截获该请求以便可以（1）确认是和该用户的CDN服务器集群（2）将客户的请求重定向到该集群的某台服务器

大多数的CDN利用利用DNS来截获和重定向请求。下面我们来解析一下这个过程。

  1. 用户访问这个页面

  2. 用户点击视频链接，此时有个对于http://netcinema.com/6Y7B23V的DNS请求

  3. 用户LDNS将该DNS请求中继到authoritative DNS server中，权威DNS服务器发现主机名中包含video，为了将此CDN请求转交给KingCDN，权威DNS服务器不返回一个IP地址，而是向LDNS返回一个KingCDN域的hostname，比如a1105.kingcdn.com

  4. 此时对，DNS请求进入KingCDN专用DNS基础设施。用户的LDNS发送第二个请求，此时对a1105.kingcdn.com的DNS请求，KingCDN的DNS系统最终返回KINgCDN的内容服务器的IP地址。

  5. LDNS向用户主机转发内容服务CDN节点的IP地址

  6. 建立TCP连接。

集群选择策略（cluster selection strategy），即动态将客户定向到CDN中的某个服务器集群或数据中心。

最简单的策略时， 地理上最为邻接（geographically closest）。除此外还有根据流量条件为客户决定最好的集群，这里采用了实时测量（real-time measurement）

套接字编程

UDP套接字编程

我们之前说过，每个进程好比一座房子，socket彷佛一扇门。

当使用UDP时，必须先将目的地址负在该分组之上。该分组传到发送发的socket周。因特网将使用该目的地址并通过因特网为该分组选录到接受进程的socket。当分组到达接收socket时，接受进程通过该socket取回分组。

过程如下

下面为python代码


# UDPClient

from socket import *
serverName = '192.168.235.128'
serverPort = 12000
clientSocket = socket(AF_INET, SOCK_DGRAM)  # AF_INET表明使用的是ipv4协议，SOCK——DGRANM表明使用的是使用UDP
message = input("Input lowercase sentence: ")
clientSocket.sendto(message.encode(),(serverName,serverPort))   # 我们的内容在发送到socket时，需要以字节进行传输
modifiedMessage, severAddress=clientSocket.recvfrom(2048)
print(modifiedMessage.decode())
clientSocket.close()

from socket import *
serverPort = 12000

    modifiedMessage = message.decode().upper()

TCP套接字编程

与UDP不同，TCP是一个面向连接的协议。所以，我们在建立连接之前需要进行TCP的三次握手过程。

首先，与UDP大致相同，TCP服务器在客户试图发起接触前必须左为进程运行起来，第二，服务器程序必须具有一个特殊的套接字，这个socket欢迎来自运行在任意主机上的客户进行初步接触。

客户进程向服务器发起一个TCP连接，这是由客户程序创建一个TCPsocket完成的，它指定服务器的ip地址和socket的端口号。生成器套接字后，该客户发起三次握手，并创建一个与服务器的TCP连接。

三次握手期间，服务器感知客户与特殊socket建立连接，于是生产一个新的socket。它是专门对客户进行连接的新的socket，称为 连接套接字（connection socket）。

过程如下

python代码如下


# scoketServer

from socket import *
serverPort = 12000
ServerSocket = socket(AF_INET,SOCK_STREAM)          # 创建服务器的套接字，第一个参数说明IPv4，第二个套接字说明TCP套接字
ServerSocket.bind(('',serverPort))                  # 将该端口与socket联系起来
ServerSocket.listen(1)                              # 对于Tcp来说ServerSocket相当于欢迎之门
print("The server is ready to receive")
while True:
    connectionSocket, addr = ServerSocket.accept()  # 聆听TCP连接请求，当client敲门时，程序为serverSocket调用accept
                                                    # 方法，这就会建立新的套接字，这个有这个特定客户独享
    sentence = connectionSocket.recv(1024).decode()
    capitalizedSentence = sentence.upper()
    connectionSocket.send(capitalizedSentence.encode())
    connectionSocket.close()
